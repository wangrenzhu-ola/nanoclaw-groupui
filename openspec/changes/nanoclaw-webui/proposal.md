## 1. 项目定位与背景

NanoClaw 当前主要通过 CLI 与第三方 IM 渠道工作，缺少原生私有化协作界面。本提案目标是为 NanoClaw 构建 Slack 风格的 AI Agent 协作 WebUI，满足“人-人-多Agent”高频协作、可审计、安全可控、可私有部署的业务需求。

**核心定位**：为 NanoClaw 打造私有化、Slack 风格的 AI Agent 与人协作交互界面。

## 2. 目标与非目标

### 2.1 目标（In Scope）

1. 平台 MUST 提供频道、私信、线程三层会话结构，并实现消息与权限隔离。
2. 平台 MUST 保留并增强现有 NanoClaw 的实时流式对话、Agent 管理、Soul 管理、任务管理能力。
3. 平台 MUST 对齐 Slack 关键交互模式：侧边栏分区、右侧线程抽屉、消息操作菜单、未读与通知体系。
4. 平台 MUST 保持私有化架构：Socket.io 流式、SQLite + 本地文件持久化、MiniMax API 接入链路。

### 2.2 非目标（Out of Scope）

1. 平台 MUST NOT 引入第三方云存储、云消息队列、云通知网关作为运行时依赖。
2. 平台 MUST NOT 实现跨工作区联邦协作与外部组织共享频道。
3. 平台 MUST NOT 在本阶段支持音视频会议、屏幕共享、电话集成。
4. 平台 MUST NOT 修改 NanoClaw Core 的业务语义，仅在兼容边界内扩展 WebUI 与 BFF。

## 3. 业务价值

1. 协作效率：线程与提及机制 SHOULD 将频道噪音降低，提升信息可追踪性。
2. 运维治理：RBAC + 审计日志 MUST 提供关键操作追溯闭环。
3. 私有安全：本地化存储与内部认证 MUST 满足企业内网合规要求。
4. 可演进性：OpenSpec 文档体系 MUST 支持需求-设计-任务的一致演进。

## 4. 范围边界

### 4.1 功能边界

- 覆盖：频道、公私私信、线程、消息操作、未读状态、通知规则、搜索、文件管理、权限审计。
- 不覆盖：外部 IM 双向桥接、跨租户账户系统、复杂计费。

### 4.2 系统边界

- WebUI + BFF 属于本提案交付范围。
- NanoClaw Core、MiniMax API 调用契约、Socket.io 通讯协议属于兼容边界，不做破坏性变更。

## 5. 术语定义

| 术语 | 定义 |
| --- | --- |
| Workspace | 一个独立私有部署实例内的协作空间 |
| Channel | 频道会话，分公开/私有，承载多人协作 |
| DM | 私信会话，支持 1v1 与 Group DM |
| Thread | 基于单条消息展开的子会话 |
| Agent | NanoClaw 注册的可执行智能体（RegisteredGroup 抽象） |
| Soul | Agent 的持久人格/规则文档（`CLAUDE.md`） |
| Sandbox | Agent 容器的挂载与运行隔离配置 |
| Mention | `@agent`、`@here`、`@everyone` 等提及触发语义 |

## 6. 技术约束与兼容原则

1. 前端 MUST 采用 Next.js App Router 体系并兼容现有组件规范。
2. 实时通信 MUST 使用 Socket.io，保持与既有事件模型兼容。
3. 数据层 MUST 继续使用 SQLite 与本地文件系统，不引入云 DB。
4. 模型调用 MUST 兼容 MiniMax API 既有链路，不新增第三方推理服务依赖。
5. 部署 MUST 支持 Docker Compose 私有化部署与容器网络互通。

## 7. 核心能力增补

1. 会话结构补全：私信（1v1/Group DM）、频道、线程统一建模。
2. 消息能力补全：编辑、撤回、引用回复、Emoji 反应、复制、转发。
3. 频道能力补全：公开/私有类型、归档替代硬删除、权限边界明确。
4. 提及与状态补全：`@here`、提及高亮、未读聚合、在线状态、静音。
5. 搜索与文件补全：全域检索、筛选器、上传预览下载与类型扩展。
6. 通知补全：全局规则、会话级定制、关键词提醒、免打扰。
7. 治理补全：RBAC、关键操作审计日志（操作者/时间/内容/IP）。

## 9. 风险评估与应对

| 风险类别 | 风险描述 | 等级 | 应对策略 |
| --- | --- | --- | --- |
| 技术风险 | SQLite 并发写导致 `SQLITE_BUSY` | 高 | 启用 WAL、短事务、写重试退避、写路径收敛到 BFF |
| 技术风险 | Socket.io 断线导致消息状态不一致 | 高 | ACK + 重放队列 + 重连补偿同步 |
| 兼容风险 | MiniMax 流式格式差异导致渲染异常 | 中 | 在 BFF 增加统一适配层与格式校验 |
| 需求风险 | Slack 对齐范围扩张导致交付失控 | 中 | 以 OpenSpec 变更单控制 scope，超范围需求走新 proposal |
| 运维风险 | 私有化备份策略缺失导致恢复失败 | 高 | 制定备份/恢复演练标准并纳入上线门禁 |

## 10. 依赖与约束

1. 依赖 NanoClaw Core 暴露稳定消息与任务数据结构。
2. 依赖本地挂载卷可读写、容器网络可互通。
3. 依赖现有认证机制（Basic Auth / NextAuth）在内网可用。
4. 依赖 OpenSpec 流程在变更前后可执行审阅与归档。

## 11. 成功标准

1. 规格完整性 MUST 达到：模块需求、BDD 场景、验收项三者一一映射。
2. 功能闭环 MUST 达到：频道/私信/线程、通知、权限、审计、部署验收可执行。
3. 兼容性 MUST 达到：不破坏既有 Socket.io、本地持久化、MiniMax 链路。
4. 可验证性 MUST 达到：所有 P0/P1 验收项具备自动化或手工可重复测试步骤。
